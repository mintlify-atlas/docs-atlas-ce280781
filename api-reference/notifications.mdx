---
title: 'Notifications API endpoints'
description: 'List review notifications, mark them as read, and subscribe to a real-time Server-Sent Events stream for live review updates in your Lynx dashboard.'
---

Lynx generates a notification each time a code review completes — whether it succeeded or failed. The notifications API lets you list recent notifications, mark individual ones or all of them as read, and subscribe to a real-time event stream using Server-Sent Events (SSE).

---

## GET /notifications

Returns the 50 most recent notifications for the authenticated user, ordered by most recent first.

**Authentication required:** yes

```bash
curl https://yourdomain.com/api/v1/notifications \
  -b "lynx_session=<your-session-cookie>"
```

### Response

```json
[
  {
    "id": 12,
    "type": "review_complete",
    "review_id": 42,
    "repo_name": "owner/my-repo",
    "pr_number": 17,
    "provider": "openai",
    "error": null,
    "read": false,
    "created_at": "2024-06-01T14:23:11+00:00"
  }
]
```

<ResponseField name="id" type="integer">
  Unique notification ID.
</ResponseField>

<ResponseField name="type" type="string">
  Notification type string (e.g., `"review_complete"`).
</ResponseField>

<ResponseField name="review_id" type="integer">
  ID of the associated review. Use this with `GET /usage/reviews/{review_id}` to retrieve full review details.
</ResponseField>

<ResponseField name="repo_name" type="string">
  Full repository name in `owner/repo` format, sourced from the linked review.
</ResponseField>

<ResponseField name="pr_number" type="integer">
  Pull request or merge request number, sourced from the linked review.
</ResponseField>

<ResponseField name="provider" type="string">
  LLM provider used for the review (`openai`, `gemini`, or `anthropic`).
</ResponseField>

<ResponseField name="error" type="string">
  Error message if the review failed. `null` for successful reviews.
</ResponseField>

<ResponseField name="read" type="boolean">
  Whether the notification has been marked as read.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the notification was created.
</ResponseField>

## PATCH /notifications/{notification_id}/read

Marks a single notification as read.

**Authentication required:** yes

```bash
curl -X PATCH https://yourdomain.com/api/v1/notifications/12/read \
  -b "lynx_session=<your-session-cookie>"
```

### Path parameters

<ParamField path="notification_id" type="integer" required>
  ID of the notification to mark as read.
</ParamField>

### Response

```json
{ "success": true }
```

Returns `{"detail": "Not found"}` if the notification does not exist or belongs to a different user.

---

## PATCH /notifications/read-all

Marks all unread notifications as read for the authenticated user.

**Authentication required:** yes

```bash
curl -X PATCH https://yourdomain.com/api/v1/notifications/read-all \
  -b "lynx_session=<your-session-cookie>"
```

### Response

```json
{ "success": true }
```

## GET /notifications/stream

Opens a persistent Server-Sent Events (SSE) connection that delivers real-time notifications as reviews complete. The connection stays open until closed by the client, and Lynx sends a keepalive comment every 15 seconds to prevent idle timeouts.

**Authentication required:** yes

```bash
curl -N https://yourdomain.com/api/v1/notifications/stream \
  -b "lynx_session=<your-session-cookie>" \
  -H "Accept: text/event-stream"
```

### Response

The endpoint returns `Content-Type: text/event-stream`. Events are delivered as SSE-formatted messages. Keepalive frames are sent as SSE comments (`: keepalive`) every 15 seconds when no events are pending.

```
: keepalive

data: {"id": 13, "type": "review_complete", "review_id": 43, ...}

: keepalive
```

<Note>
  Use the `-N` flag with curl to disable output buffering when testing the stream. In production clients, use the browser's `EventSource` API or an SSE client library.
</Note>

The response headers include `Cache-Control: no-cache` and `X-Accel-Buffering: no` to ensure compatibility with nginx-proxied deployments.
